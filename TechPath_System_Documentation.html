01 System Overview

TechPath Expert is a rule-based expert system designed to recommend IT career paths for Malaysian students and professionals. The system uses Forward Chaining inference with 135 production rules to match user profiles — comprising interests, skills, and personality traits — against 18 established IT career paths in the Malaysian technology industry.

Inference Method: Forward Chaining

The system applies Forward Chaining (data-driven reasoning), starting from user-provided facts (assessment answers) and firing applicable rules until a ranked list of career recommendations is produced. This is the standard reasoning strategy used in production rule-based expert systems.

ComponentDescription
Knowledge Base18 IT career profiles with skills, salary, certifications, and progression paths
Inference EngineForward Chaining with 135 weighted rules (Primary +30, Supporting +15, Rule-3 +12, Rule-4 +10, Excellence +20, Penalty Tier ≤2/≤1, Tiebreaker, Synergy, Weak Signal +5)
Working MemoryUser assessment scores stored in browser localStorage (minat, kemahiran, personaliti)
User Interface5-step web-based assessment (HTML/CSS/JavaScript)
OutputTop 3 ranked career recommendations with detailed roadmaps

Tie-Breaking Mechanism

If two or more careers end with the exact same final score (e.g. a fully neutral profile where every answer is 3, producing 0 fired rules for all 18 careers — see Documentation Section 06, TC-03), the system does not default to a fixed "favourite" career. A random tiebreaker value is attached to each career before sorting, so the displayed Top 3 varies between attempts rather than always favouring whichever career happens to be listed first in the code.

07 Examiner Q&A Preparation

The questions below are anticipated based on the actual design decisions, bugs, and trade-offs made while building this system. Each answer is grounded in something demonstrable in the live system — point to the relevant page/section when answering live.

A. Core Expert System Concepts

Q: What type of Expert System is this, and why?
A: A rule-based Expert System using Forward Chaining. We start from known facts (the user's 45 assessment answers) and fire every matching rule to derive new conclusions (career scores), rather than starting from a goal and working backward (Backward Chaining). Forward Chaining suits this problem because we have a fixed, complete set of input facts upfront and want to explore all possible conclusions — not test one specific hypothesis.
Q: What are the main components of your Expert System?
A: Five classic ES components are present: (1) Knowledge Base — 18 career profiles + 135 rules in processing.html; (2) Working Memory — the user's answers stored as facts in localStorage (minat, kemahiran, personaliti); (3) Inference Engine — the Forward Chaining loop that fires rules and accumulates scores; (4) Explanation Facility — the "Why This Career?" panel on the results page; (5) User Interface — the 5-step assessment flow plus the results dashboard.
Q: Where is your Knowledge Base, and what does it contain?
A: Two parts: the rule base (135 IF-THEN rules in processing.html) and the career knowledge base (18 IT career profiles in knowledge-base.html and result.html, each with description, required skills, certifications, salary ranges, and career progression — sourced from verified Malaysian industry reports, see Section 05 References).

B. Inference Engine & Rule Design

Q: Walk us through what happens after a user submits the assessment.
A: (1) The three assessment pages save answers to localStorage; (2) processing.html loads those answers and runs all 135 rules against them, accumulating a weighted score per career and logging every fired rule; (3) all 18 careers are ranked by score and saved in full (not just the top 3); (4) the user is redirected to result.html, which reads the ranked list, the fired-rule log, and renders the score ring, the Explanation Facility, and the Full Inference Engine Results table.
Q: Why does your system have 8 different rule tiers (Primary, Supporting, Tertiary, Quaternary, Excellence, Synergy, Weak Signal, Penalty) instead of one flat rule set?
A: Each tier captures a different strength of evidence. Primary rules (+30) require strong interest AND skill together — the strongest single signal. Supporting/ Tertiary/Quaternary add weaker, more specific corroborating evidence. Excellence Bonus (+20) only rewards a genuinely perfect (5/5) profile in a career's most critical area. Weak Signal (+5, ≥3 threshold) exists specifically so that moderate/neutral answers still produce some differentiation instead of every career tying at zero. Penalty rules subtract points when a critical skill is weak, so a career is never recommended purely because nothing was checked.
Tip: If pressed for a number, state clearly: 18+18+18+18+5+10+18 = 105 positive-scoring rules, 23 penalty conditions, 7 tiebreaker rules = 135 total.
Q: Why do some rules use ≥3, others ≥4, others ≥5 as the threshold?
A: The threshold reflects how strong the evidence needs to be for that tier. ≥4 is the standard "agree" bar used by Primary/Supporting rules. ≥5 ("strongly agree" / perfect skill) is reserved for Excellence Bonus and a handful of Quaternary rules that should only reward truly exceptional answers. ≥3 ("neutral/slightly agree") is used only in the Weak Signal tier, deliberately set low so moderate profiles are not scored identically to someone who disagreed with everything.

C. Conflict Resolution & Certainty

Q: Multiple rules can fire for multiple careers at once — how does the system decide the final answer?
A: We use Additive Weighted Scoring as our conflict resolution strategy (see Section 03 of this document and the "How The System Resolves Conflicts" section on about.html). Every rule that fires adds its weight to that career's running total — we do not stop at the first match, and we do not use a fixed priority order. The career with the highest cumulative score after all 135 rules have been evaluated wins. This was chosen over First Match and Priority Ordering specifically because it rewards consistent, multi-dimensional evidence rather than any single strong signal.
Q: How does the system express uncertainty when the match is weak?
A: Through a Confidence Level (our Certainty Factor implementation) shown directly above the rule list in "Why This Career?". It has three bands — Strong Match (≥65% and ≥4 rules fired), Moderate Match (≥40%), and Exploratory Match (below 40%) — each with a different explanatory message. Critically, the badge is derived from the exact same percentage shown in the score ring, so the two numbers can never contradict each other (this was a real bug we found and fixed — see Test Case TC-06).
Q: What happens if a user answers every question identically (e.g. all 5s)?
A: Every rule with a satisfied threshold fires simultaneously, so the differentiator becomes how many rule tiers are structurally attached to each career — not the answers themselves. Only 5 of 18 careers carry an Excellence Bonus rule, so those 5 always rise to the top; among them, AI Engineer and Data Scientist win because they also hold the two highest-value Synergy rules (+10 and +8). This is documented and verified as Test Case TC-05.

D. Testing, Validation & Limitations

Q: How did you test and validate the inference engine?
A: Through manual trace-based boundary value testing (Section 06 of this document) — since every rule branches on a fixed numeric threshold (≥3/≥4/≥5/≤2/≤1), we tested the extremes (uniform inputs of 1, 2, 3, 4, and 5) plus key edge cases (empty academic field, score-display consistency, penalty symmetry). 9 test cases were run; 3 surfaced real defects, which were fixed in the codebase and re-verified.
Q: Can you describe a bug you found and how you fixed it?
A: Yes — testing with all-1 answers revealed that 4 careers (AI Engineer, Data Scientist, Machine Learning Engineer, Cloud Engineer) had no penalty rule at all, so they "won" by default whenever the user's profile was weak everywhere, even though nothing in the answers actually supported those careers. We fixed this by adding Rules 122–135, giving every one of the 18 careers a penalty path at both a moderate (≤2) and severe (≤1) threshold, then re-verified with the same test inputs.
Q: What are the limitations of your system?
A: (1) Data persistence — results are stored in browser localStorage, not a server database, so they do not sync across devices and are lost if the browser storage is cleared; (2) No Knowledge Acquisition Module — rules are hardcoded in JavaScript rather than editable through an admin interface, so updating the rule base requires a developer; (3) the rule thresholds and point weights were designed using domain knowledge of real IT job requirements, not derived from a statistically validated dataset — this is reasonable for an academic prototype but would benefit from real user outcome data in production.
Q: How would you improve this system given more time?
A: Three priorities: (1) a Knowledge Acquisition Module — an admin panel to add/edit rules without touching code; (2) move from localStorage to a proper backend (e.g. Firebase or a small REST API) for persistence across devices and sessions; (3) replace the manually-designed point weights with weights calibrated against real graduate employment outcomes, turning the Certainty Factor into a statistically grounded confidence score rather than a threshold-based heuristic.
